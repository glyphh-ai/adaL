# Recall benchmark — pre-registered protocol

*Registered before any results. The question is existential, so the
design is adversarial toward Ada's current recall: it targets exactly
the queries lexical matching should fail.*

## The question

When a query reaches Ada's **free-text recall path** (`recall` /
`think` / the `ask` fallback — set-cosine over tokens, NOT the exact
op engine), how reliable is retrieval, and does an **embedding** arm
(MiniLM) fix the cases lexical matching misses — without regressing the
cases it already handles?

This path matters because a human or naive client asking a question in
prose hits lexical recall, not the structured `query` ops that scored
92–100% in Phase 2/3. Those ops require the caller to formulate
`count`/`who`/`lookup` etc. The natural `ask` does not.

## Methods (the manipulated variable)

1. **lexical** — Ada's shipped `score_thought` (token set-cosine over
   content + slots). The status quo.
2. **embedding** — `all-MiniLM-L6-v2`, normalized cosine over the fact
   sentence (the same model used as the Phase 2 RAG baseline).
3. **hybrid** — reciprocal-rank fusion of (1) and (2), k=60. No scale
   tuning, no per-set weights.

All three rank the SAME corpus of fact sentences for each query. The
only difference is the scorer.

## Corpus

Per seed: 60 personas, each with 6 facts (residence, job, employer,
favorite color, a hobby, a vehicle) → 360 fact sentences. Generated
deterministically from fixed templates.

## Query set — six types, the independent variable

Query phrasings are generated by an LLM (Haiku) **once** and frozen to
JSON, so the wordings are not hand-chosen by the author and the run is
reproducible. Each query has exactly one correct target fact (or none,
for negatives).

1. **direct** — shares the fact's keywords ("what city does X live in?"
   vs "X lives in Denver"). Control: lexical should be strong;
   embedding must not regress here.
2. **paraphrase** — same proposition, reworded, reduced lexical overlap
   ("where is X based?").
3. **conceptual** — asks via a synonym/category ("what does X do for
   work?" for `works_as`; "X's ride" for vehicle).
4. **indirect / two-hop** — answer requires the entity bridge Ada's
   `ask` already attempts ("who does X's employer pay?" style).
5. **contextual** — a non-question context line (the hook use case)
   whose relevant fact shares meaning, minimal tokens ("planning a
   surprise in X's home town" → the residence fact).
6. **negative** — no fact is relevant (a plausible but absent
   attribute). Measures false injection.

Target ≥ 40 queries per type per seed.

## Metrics

Positives (types 1–5):
- **hit@1**, **hit@3** — is the target in the top-1 / top-3?
- **MRR** — mean reciprocal rank of the target.

Negatives (type 6) and the hook use case:
- **false-injection rate** at an injection threshold τ — fraction of
  negative queries whose top-1 score ≥ τ (would wrongly inject).
- For the substrate framing: the **flat-file baseline** is "every fact
  always in context" → hit = 1.0 always, injection = N/A; its cost is
  unbounded context, not retrieval misses.

All reported per method × per query-type, mean ± std over **5 seeds**,
with the lexical→best Δ (effect size), not just significance.

## Win / kill conditions

- **Embedding/hybrid WINS** if, vs lexical: hit@3 improves by **≥ +10
  points** averaged over paraphrase + conceptual + contextual, **and**
  does not regress direct by more than 2 points, **and** false-injection
  at the operating τ is no worse.
- **KILL (lexical is enough)**: < +5 points on the fuzzy types → the
  embedding arm does not ship; we state plainly that Ada's recall is
  lexical and is reliable only for keyword-overlapping queries, and the
  product answer is "drive the exact op engine for everything else."
- Between +5 and +10: ships as an opt-in recall mode, recorded weak.

## What a poor result would mean (stated up front)

If lexical recall cannot retrieve paraphrased/conceptual queries and
embeddings don't rescue it, then Ada's natural-language `ask` is
unreliable for anything but keyword-matching, and the honest product
position narrows to: **Ada is an exact structured store driven by an
LLM that formulates ops** — not a fuzzy semantic search engine. We
report that outcome if that is what the numbers say.
